## DD-2 — Objective skip-gate: substance rule + dispatch predicate, no bypass
- **content:** Per-skill objective "skip allowed IFF X" rules live in skill substance, paired with a predicate handler (`evalDispatchPredicate` inside `essense-flow-tools.cjs evaluatePredicate`) that reads dispatch-count fields from output-artifact frontmatter and refuses the phase transition when the count is below threshold AND the skip condition is unmet. New phrase recognizers: `with sufficient lens dispatch`, `with sufficient verifier dispatch`, `with sufficient sub-architect dispatch`. NO bypass flag exists. (Later precedent hardening, DD-13: even "harness lacks the Agent tool" is not an admissible skip cause — the build halts and surfaces a SURPRISE instead of substituting.)
- **why:** Root A of the closure plan, "master-judgment-condense" — the orchestrator decided per-session that "scope justifies a condensed path" and silently skipped sub-agent dispatches. Symptoms: drift-8, drift-7, condensed sprint-2/3 paths in tmp-spike-S10.5, doc-drift recurrence-pattern row 11. Subjective judgment kept eroding the dispatch discipline, so the criterion was made objective and machine-enforced.
- **source:** `redesign/06-decisions.md:1177` (Root A context at :1175; bypass-none precedent reaffirmed in DD-13 at `tmp-spike-CLOSURE/.pipeline/elicitation/SPEC.md:185-187`)
- **proposed inline principle:** Phase-exit gates must be objective and machine-checked: each skill states a concrete "skipping sub-agent dispatch is allowed only if X" rule, and the gate tool counts actual dispatches recorded in the output artifact and refuses to advance when the count is below threshold and the skip condition is unmet. There is no bypass flag and no "environment can't do it" escape — if the gate blocks wrongly, fix the rule or halt and surface the problem; never route around the gate.
- **keep/fold/drop:** **keep** — core containment thesis of the whole plugin; survives all four rebuild changes unchanged (the counting source may move from frontmatter to disk artifacts under artifacts-authoritative state, but the principle is the principle).

## DD-12 — CLI-first orientation: documented ops for deterministic work, with inclusion/rejection gates
- **content:** Six sub-clauses. **(a)** Every architect (master + sub-architect) brief carries the rule: for every AC and task spec authored, decide whether a CLI op is the right implementation surface; if yes, spec it per the (b) standard; if no, record one of four rationales (judgment-dependent / user-interaction-required / trivial-file-edit-no-validation-needed / not-deterministic-enough). Per-AC cost-benefit recorded in the task spec. **(b)** Mandatory 8-section per-op spec at `redesign/cli-spec/ops/<op-name>.md`: 1 purpose (cites the AC/DD it operationalizes), 2 full invocation shape, 3 every argument enumerated (name, type, required/optional, default, purpose, example, validation), 4 every exit code with semantic meaning, 5 output schema (typed JSON fields or line-by-line text semantics), 6 failure modes + stderr diagnostics + exit-code mapping, 7 ≥2 worked examples (happy path + ≥1 failure path, with expected stdout/stderr/exit code), 8 cross-references. **(c)** An op enters scope IFF ≥1 inclusion criterion (replaces deterministic-LLM-task; replaces cumbersome multi-step yaml edit/manual cross-reference; adds new capability; enables external integration via shell-pipeable exit codes) AND zero rejection criteria (pure 1-line wrapper with no value-add; slower than direct without justifying validation; dependency/weight without proportional value; duplicates an existing op; can't articulate the cumbersome thing replaced in one sentence). **(d)** Existing round-1 ops grandfathered. **(e)** Scope-only — names no specific ops; architect surfaces them. **(f)** MCP server explicitly out of scope per user verdict "i do not care about the mcp thing".
- **why:** User observation 2026-05-12 (verbatim in source): Claude "is great at using CLIs"; fold in CLIs that are "very well documented with clear instructions per call, item, argument purpose etc.", placed "only where they can really help, not add things that complicate and make us slower or heavier". Deterministic shape-checks done by LLM judgment were slow, expensive, and imprecise.
- **source:** `tmp-spike-CLOSURE/.pipeline/elicitation/SPEC.md:147-178` (sub-clauses: a@149, b@151-159, c@161-172, d@174, e@176, f@178)
- **proposed inline principle:** Move deterministic, judgment-free work out of the model and into CLI operations — but only where the op replaces something genuinely cumbersome: it must replace a zero-judgment model task, a cumbersome multi-step edit, add a capability, or enable shell-pipeable integration, and it is rejected if it is a thin wrapper, slower than direct use, dead weight, a duplicate, or its purpose can't be stated in one sentence. Every op ships fully documented: purpose, complete argument list with types and validation, every exit code with meaning, output schema, failure modes with diagnostics, and at least two worked examples including a failure path.
- **keep/fold/drop:** **keep** — clauses (a)(b)(c) are durable, self-containable engineering discipline central to the rebuilt plugin. Drop clauses (d) (grandfathering — moot after rebuild), (e) (scope bookkeeping), (f) (one-off MCP exclusion verdict).

## DD-15 — Stepwise step emission across all 6 skills; explicit advance, idempotent replay
- **content:** Ship `essense-flow-tools next-step --skill <name> --cursor <path>` for ALL six skills (elicit, research, architect, build, review, verify) — user verdict "All 6 skills"; partial rollout rejected. Each SKILL.md body parses into ordered heading-bounded steps; the CLI emits one step's substance per call; `cursor.yaml` tracks `skill` + `step_index` + `step_emitted_at`. Replay: re-calling re-emits the current step (idempotent). Auto-advance on emit is REJECTED — the master must explicitly call the separate `step-advance` op after the step's substance closes, preserving the "master decides step is done" gate. Predicate gain: `step_index == total_steps` invariant at skill-complete joins the cursor/phase consistency audit.
- **why:** Failure mode: master loads the whole SKILL.md body up front; later-step substance drifts under accumulated earlier-step context — empirically observed as "6 versions of master-context-drift across long pseudocode (v0.6→0.11 the closure exists to fix)". Step-at-a-time emission lets consumed steps leave context.
- **why-rejection detail:** build-only pilot rejected because partial coverage seeds inconsistency across the skill set and defers the drift fix for the other 5 skills; rejecting entirely would itself be the judgment-condense pattern at the elicit layer.
- **source:** `tmp-spike-CLOSURE/.pipeline/elicitation/SPEC.md:354-358` (NOTE: a different, unrelated "DD-15" — `sprint_iteration` state field — exists at SPEC.md:471 and `redesign/06-decisions.md:1481`; this entry is the round-9 stepwise decision)
- **proposed inline principle:** Deliver long multi-step procedures to the model one step at a time: a tool emits the current step's text and a cursor records the position, so earlier steps fall out of context once consumed and later steps don't drift under accumulated prompt weight. Re-requesting the current step re-emits it unchanged, and the cursor never auto-advances — the orchestrator must explicitly mark a step done before the next step is emitted, and a skill counts as complete only when the cursor has consumed every step.
- **keep/fold/drop:** **keep** — directly addresses context-drift, the rebuilt plugin's central constraint; applies uniformly (no partial rollout) and is independent of which schema/state mechanism backs the cursor.

## DD-17 — Section-at-a-time validated writes; refuse-on-violation, never warn-and-write
- **content:** Atomic structured-output emission (section-write op family for ARCH.md / SPEC.md / task specs) — architect decides the exact op family, under three binding constraints from elicit: (1) any section-write op MUST refuse on schema violation with a diagnostic — soft-warn-and-write REJECTED at elicit because it defeats blast-radius reduction; (2) whole-document-write may co-exist during migration, architect picks the deprecation timeline with smoke-fixture parity protection; (3) the ops land in the same single sprint as the sibling decisions. Architect must evaluate each op against the DD-12 inclusion/rejection criteria with recorded rationale.
- **why:** Blast-radius failure mode: master emits a ~30KB ARCH.md block in one shot; a schema violation at byte 25K wastes all prior emitted work plus the main-context tokens spent producing it. Section-at-a-time emission contains the damage to one section.
- **source:** `tmp-spike-CLOSURE/.pipeline/elicitation/SPEC.md:367-372` (failure-mode statement at :241; NOTE: an unrelated hotfix "DD-17" — `<n>` substitution diagnostic — exists at SPEC.md:481)
- **proposed inline principle:** When the model writes structured artifacts, prefer section-at-a-time emission through a validating writer over one-shot whole-document writes: the writer checks each section against the schema and refuses with a diagnostic on violation, so a schema error costs one section instead of the whole document. Write-then-warn is forbidden — a write that lands despite failing validation defeats the entire containment.
- **keep/fold/drop:** **fold** — into **Phase 1 schema single-source**: once one canonical schema generates all artifact shapes, the validating section-writer is the natural write path. The kernel to carry: validate-before-write, refuse-with-diagnostic, bounded blast radius.

## DD-18 — Conservative defaults: explicit args required, no ambient-state inference
- **content:** Short-string CLI ops accept ONLY the description string; ALL other fields (`target_phase`, `source_artifact`, `source_anchor`, `closure_criterion`, `sprint`, `round`, `skill`) require explicit `--field=value` flags. NO inference from `cursor.yaml` or `state.yaml`. No echo-back confirm step (nothing inferred, nothing to confirm). The CLI rejects missing required fields with a diagnostic listing the missing flags. Binding constraint: architect MAY add an opt-in `--from-cursor` inference flag for ergonomics, but default behavior stays explicit-args-required, and any opt-in inference MUST echo every inferred field in the op output for the audit trail.
- **why:** Failure mode: "magic" inference surprises the master when ambient context (e.g., `cursor.yaml skill`) doesn't match the op's intended target — the master's mental model of state diverges from the CLI's inferred state, producing silent drift. Verbose flags accepted as the cost: "explicit substance over inferred substance" mirrors the no-bypass gate philosophy. Echo-back-confirm rejected as round-trip cost that doesn't remove the surprise surface; silent aggressive inference rejected as higher drift risk; defer-to-architect rejected because explicit-vs-magic is a user-stake discipline question.
- **source:** `tmp-spike-CLOSURE/.pipeline/elicitation/SPEC.md:374-378` (NOTE: an unrelated hotfix "DD-18" — version source-of-record reconciliation — exists at SPEC.md:487)
- **proposed inline principle:** Tool operations never infer argument values from ambient session state: every field beyond the primary input requires an explicit flag, and missing required fields fail with a diagnostic listing exactly which flags are absent. If an opt-in convenience flag fills values from context, the operation must echo every inferred value in its output so the record shows what was actually written — silent inference lets the caller's mental model and the tool's actual writes drift apart.
- **keep/fold/drop:** **keep** — durable interface-design rule, self-contained, and exactly the kind of explicitness a public audience needs spelled out.

## DD-19 — Stale claims: one recovery surface (heal), no in-band timers, audited staleness
- **content:** NO auto-release timer on register pop semantics; NO standalone `register-unclaim` op. Stale-claim handling routes exclusively through the `/heal` skill: heal sweeps the register for items with `status: in_progress AND claimed_at > threshold` (threshold architect-picked; candidate 24h default + per-skill override), then surfaces stale claims to the user per item OR auto-releases as a batch with an audit-trailed reason. Audit-substance extension: drift-audit gains a new staleness check ("no register item in_progress with claimed_at older than threshold" — landed as drift-10). Trade-off accepted explicitly: coverage cadence couples to /heal invocation cadence; architect may propose heal-on-phase-transition triggers, but the mechanism stays heal-mediated, never in-band per-op. Applies even if pop semantics aren't adopted (in_progress states arise via other paths).
- **why:** Stale-claim leak breaks the coverage guarantee: pop marks an item `in_progress`; the model's context closes before mark-done; the item stays claimed forever; subsequent pops skip it; the "every queued item touched once" invariant is silently broken — "if pop-based register has no stale-claim handling, 'coverage guarantee' is theatre" (SPEC.md:269). In-band timers rejected for per-op state-machine complexity (claim timestamps + sweep-on-read + races); manual unclaim rejected for duplicating heal's responsibility and leaking coverage logic across two surfaces.
- **source:** `tmp-spike-CLOSURE/.pipeline/elicitation/SPEC.md:380-384` (theatre quote at :269)
- **proposed inline principle:** Any "claimed / in-progress" marker on a work item needs exactly one recovery surface: a recovery sweep that finds items claimed longer than a threshold and either surfaces each to the user or releases them with an audit-trailed reason. Do not build in-band auto-release timers or scatter unclaim operations across the toolset, and add an audit check asserting no item stays claimed past the threshold — without enforced staleness handling, any "every queued item gets handled" guarantee is theatre.
- **keep/fold/drop:** **fold** — into the same **Phase 3 unknowns-ledger** principle as DD-10 (heal owns ledger staleness; the audit check rides along). The register-specific mechanics go; the single-recovery-surface + audited-staleness kernel stays.
