## M-2 — Engine-source citation requirement for prescribed pseudocode
- **content:** Substance rule for architect + sub-architect contract: prescribed pseudocode that references **engine output strings, library error classes, library API behavior, or tool-scanner rules** MUST cite the engine source at `<file>:<line>` (and version SHA if known). If the author cannot cite a source, the task's agency level downgrades from `prescribed` to `guided` (agent gets latitude instead of literal pseudocode). Enforcement predicate: any task spec with `agency_level: prescribed` whose pseudocode matches `<library/engine> (throws|emits|returns|produces)` must contain a `<file>:<line>` citation within 5 lines, scanned at task-spec-write time (landed as Sprint 10 T-1002, exit code 19 on violation).
- **why:** Closes RC-β + RC-δ — pseudocode authored "in vacuum." Three concrete incidents: (a) round-4 D-Rd4-X: pseudocode named `YAMLException` from upstream js-yaml docs, but the **vendored** js-yaml under `noRefs: true` config empirically throws `RangeError 'Maximum call stack size exceeded'` — a different class; (b) D-Rd4-Y: pseudocode prescribed Unicode `≤` regex while the actual engine at `audit-substance-frozen.cjs:144` emits ASCII `<=` — authored without reading the engine line; (c) round-12 T-979: a decision assumed an env-var hook covered legacy step-advance; grep would have shown the hook wired only into the new-schema paths (`tools.cjs:2593-2605` vs `:2277`).
- **source:** `redesign/META-GAP-ROUND-LOOP.md:80-86` (mechanism); `:54-55` (D-Rd4-X/Y incidents); `redesign/SPRINT-10-CLOSURE-FINAL-PLAN.md:46` (T-979 incident), `:90` (T-1002 enforcement).
- **proposed inline principle:** A design may not assert how an external thing behaves — what a library throws, what a tool emits, what a scanner accepts — unless the author read the actual source or observed the actual behavior, and cites where. Docs for the upstream project are not evidence about your vendored, configured copy. If the behavior cannot be verified at design time, say so explicitly and hand the executor judgment latitude plus the named unknown, instead of false precision.
- **keep/fold/drop:** **KEEP — and the rewrite must preserve the full ORIGINAL scope.** Original intent precisely: (1) four claim categories — engine output strings, library error classes, library API behavior, tool-scanner rules — covering **both project substrate and third-party/vendored libraries**; (2) version pinning (SHA if known), because behavior is config- and version-specific (the RangeError incident was a *config-dependent* deviation, `noRefs: true`); (3) a defined fallback — downgrade agency to guided + surface the unknown as a ledger entry, never silently proceed. If the rule is narrowed to existing-substrate-only citations, the vendored-library and version-pinning halves MUST reroute into the `unknowns[]` ledger (mechanism 3): "library X under config Y — behavior unverified" becomes a structured unknown surfaced to the user, not a dropped requirement.

## M-3 — Cross-module-concern surfacing checklist at synthesis
- **content:** Master architect's synthesize step adds an explicit checklist: for each module pair (M_i, M_j) ask (1) does M_i output have a shape M_j consumes? (data dep) (2) does M_i artifact reference M_j artifact path? (3) does M_j output depend on M_i output's substance (e.g. string vocabulary)? If yes to any: surface as a cross-module concern + close it in the decision ledger BEFORE packing the sprint. Cost ~N(N-1)/2 pairs. Verifiable check: ≥1 CMC entry per module pair where the architecture's seams table shows a data dependency.
- **why:** Closes RC-γ — sub-architects work in module silos; master's synthesis empirically caught only ~half the cross-module concerns; the rest surfaced downstream at build/review/verify (e.g. CMC-Rd4-3: M4's fixture-runner had to recognize 5 kind-strings M1 declared three rounds later).
- **source:** `redesign/META-GAP-ROUND-LOOP.md:88-99`; RC-γ at `:64`; CMC-Rd4-3 incident at `:57`. Landed as Sprint 10 T-1003 (`SPRINT-10-CLOSURE-FINAL-PLAN.md:91`).
- **proposed inline principle:** When a design is split across parallel designers, the integrator must walk every pair of modules and ask: does one consume the other's shapes, paths, or vocabulary? Every yes is recorded and resolved before any task is dispatched — integration seams found at build time cost a full amendment round each.
- **keep/fold/drop:** **KEEP.** Architecture-independent, public-ready, cheap. Under single-source schema (rebuild 1) the shape-consumption half becomes partially mechanical (generated schemas can be diffed), but path-reference and vocabulary deps remain judgment checks.

## M-4 — Build-phase pre-flight reality probe
- **content:** Task agent at dispatch start runs a reality probe: workspace state checks (is this a git repo? does this command resolve?), dependency availability (libraries present? versions?), tool-scanner-vs-content compat (will the scanner reject what the task must write?). Probe output persists to the completion record as `preflight_probe:`; a blocker halts the task EARLY with a diagnostic instead of failing mid-execution. Sprint 10's ratified execution protocol promoted this to a master-side pre-dispatch step: `ls` every write target, grep every cited symbol, `node --check` every cited script, results to `.pipeline/.preflight/<task-id>.json`, probe failure = halt dispatch.
- **why:** Closes RC-β indirectly — pseudocode-vs-reality drift caught at dispatch, not at execution-against-broken-assumption. Incident: round-4 D-Rd4-Z — task T-rd2-m1-014 prescribed `git restore --source <sha>` but the workspace had no git ancestor at any level; agent discovered this only at task pre-flight and returned BLOCKED, consuming a full amendment round.
- **source:** `redesign/META-GAP-ROUND-LOOP.md:101-107`; D-Rd4-Z at `:56`; protocol promotion at `redesign/SPRINT-10-CLOSURE-FINAL-PLAN.md:226-232` (Step 1) and `:339` (wrong-order reversal table).
- **proposed inline principle:** Before an agent starts work, verify the assumptions the work stands on: the paths exist, the symbols cited are really in the code, the tools resolve, the environment matches what the plan assumed. A blocker found before work starts costs a halt; one found mid-work costs a redesign round.
- **keep/fold/drop:** **FOLD into rebuild mechanism (3).** Research-first IS the probe moved to its proper place — the agent researches its substrate before acting, and anything unverifiable lands in the `unknowns[]` ledger. Keep the early-halt semantics: a probe failure surfaces to the user, never a mid-task discovery.

## L-7 — Rule-completeness lens
- **content:** New review lens (`essense-flow-rule-completeness-lens`), dispatched as the 7th parallel lens every review round. Walks every executable spec rule (rules carrying a machine-checkable `applies_to` block: kinds `regex | absence | xref | paired-xref | unchecked-rule`, closed list), runs a sweep op per rule across project scope, emits a finding per sibling violation. Every sweep hit gets re-validated by a per-candidate validator agent in clean context (verdicts: `confirmed_sibling | intentional_exception | false_positive | needs_context`). Budget: max 50 rules/round, 30s timeout/rule; overflow emits `sweep_partial: true`, never silent skip. Bootstrap baseline: first run on a mature project tags all current violations `pre-existing-acknowledged`; subsequent runs surface only new ones.
- **why:** External Unity-project evidence (2026-05-16 chat): Sprint 3 ran 4 patch rounds, review reached round 8; 5 of 6 confirmed criticals were pre-existing bugs in the SAME module surfacing one-per-round; a no-try/catch rule was patched in two classes across two rounds while a third class (ORCAGPU) was missed both times. Root cause: "the framework treats findings as singletons. It does not ask 'this rule violation — what is its family across the codebase?' before declaring the round done."
- **source:** `round-loop-closure/SPEC.md:7-19` (problem), `:29` (AC-5), `:60-78` (applies_to kinds + verdicts), `:100-107` (budget caps); `round-loop-closure/PLAN.md:108-111` (lens definition); registered agent: `plugins/essense-flow/agents/essense-flow-rule-completeness-lens.md`.
- **proposed inline principle:** When review confirms a rule violation, sweep the whole codebase for the rule's family before closing the round — one confirmed instance of a pattern almost always has unpatched siblings. Rules must therefore be written as machine-checkable patterns with scopes, not prose, and every mechanical hit gets human-grade re-validation before it counts.
- **keep/fold/drop:** **KEEP.** One of the two highest-value transferable mechanisms in the corpus. The closed `applies_to` kind list and the sweep-then-validate split both survive the rebuild intact; with single-source schema, rule encoding validation becomes part of the generated-schema check.

## L-8 — Pattern-debt lens
- **content:** New review lens (`essense-flow-pattern-debt-lens`), dispatched as the 8th parallel lens every review round. Re-runs prior-round QA-report rule sweeps against the *current* codebase; emits recurrence-findings for NEW hits not in the prior round's resolved set. Budget: max 20 prior rounds replayed; overflow surfaces `sweep_partial`. Depends on QA reports persisting per-finding `sweep_pattern` + verdict (CMC-RLC-4) and stable finding ids (CMC-RLC-3: e.g. `<rule-id>-<file-path-hash>-<line>`).
- **why:** Catches patterns that re-emerge when a patch lands but accidentally reintroduces the same shape elsewhere — the Unity evidence showed pre-existing bugs surfacing only when patches invalidated incremental compile caches; without replaying prior sweeps, a fixed family can silently regrow.
- **source:** `round-loop-closure/SPEC.md:30` (AC-6), `:104` (budget); `round-loop-closure/PLAN.md:113` (definition); CMC-RLC-3/4 at `round-loop-closure/decisions.md:49-50`; registered agent: `plugins/essense-flow/agents/essense-flow-pattern-debt-lens.md`.
- **proposed inline principle:** Every review round replays the previous rounds' confirmed-pattern sweeps against the current code, flagging only new hits — fixing a pattern once does not prevent it re-entering, and only a replayed sweep notices. This requires findings to persist their search pattern and a stable identity so "new vs known" is computable.
- **keep/fold/drop:** **KEEP.** Pairs with L-7; together they are the "finding → family → recurrence watch" pipeline. Stable finding identity folds naturally into the single-source schema work.

## INST-13 — fewest-sprints / no-budgets principle
- **content:** Verbatim: "we don't have budgets, we don't specify what needs to be done in how many turns — we have as many as we need but always aim for **the lowest amount of sprints necessary**, and we give the context necessary and no more than that. Not predetermined amounts of agents, not budgets — clean and good work without unnecessary steps." Applied across skills: no concurrency cap on wave dispatch; no cap on elicitation rounds (loop ends when threads close, not when a counter expires); no cap on heal walk-forward steps; default one sprint one wave, split only on real data-dependency or file conflict, theme-based splits rejected. One of the 5 canonical principles every skill cites (with Graceful-Degradation, Front-Loaded-Design, Fail-Soft, Diligent-Conduct).
- **why:** Anti-pattern reaction: predetermined budgets ("5 agents," "3 rounds max") either truncate real work or pad trivial work; the meaningful disciplines are minimal structure (fewest sprints) and minimal sufficient context, with length treated as signal not violation. Note the deliberate tension with M-5: M-5's round budget caps *re-opening of closed design*, not the size of honest first-pass work — the two coexist.
- **source:** verbatim text at `redesign/skill-substance/architect.md:208-212`; principle citation at `:237`; per-skill applications at `redesign/skill-substance/build.md:189`, `elicit.md:142`, `heal.md:230`; canonical-principle membership at `redesign/02-history-of-failure-modes.md:73`.
- **proposed inline principle:** Do not pre-budget effort — no fixed agent counts, turn limits, or round caps on first-pass work; take as many steps as the work needs while aiming for the fewest delivery cycles necessary and giving each worker the context it needs and no more. A long loop is information about scope, not a violation. (Distinct rule: *re-opening finished design* IS budgeted — see the two-round amendment cap.)
- **keep/fold/drop:** **KEEP.** Core philosophy, fully self-contained, public-ready. In the rebuild it should be stated once and inlined where cited (rebuild 4: codename inlining — the name "INST-13" must not survive into public text).

## BS-4 — Write-op vs shape-validator asymmetry (pattern-library shape)
- **content:** Pattern definition (field-to-ship pattern library): for each CLI write op that enforces a type contract (e.g. `parsePositiveIntOrNull`, `parseNonNegInt`, action whitelist), identify the state-file field it writes, find the shape validator, and confirm the validator enforces the SAME contract on read. Asymmetry = a direct YAML edit can introduce values the write op would reject; downstream consumers break silently. Evidence required: quote the write op's type-check line + the validator's check (or absence) for that field, both with file:line.
- **why:** Incident chain: v0.13.1 Fix-3 found the `sprint` field write-validated but read-unvalidated; a v0.13.2 audit then swept the pattern family (the L-7 idea applied manually) and found the SAME asymmetry across 10 peer fields (wave, 3 round fields, 6 ISO-timestamp fields) — direct YAML edits could slip `wave: "abc"` or `round: -1` past the validator. F1 closed the whole class: validator mirrors every write-side parser, type-mismatch surfaces as a shape error naming the offending field in dotted form.
- **source:** pattern definition at `~/.claude/skills/field-to-ship/patterns.yaml:107-122`; incident + class closure at `redesign/06-decisions.md:1546-1556` (F1, "Closes BS-4 asymmetry pattern across 10 fields") and `:1575`.
- **proposed inline principle:** Every contract enforced when a value is written must be enforced again when it is read — otherwise any out-of-band edit (a hand-edited file, a different writer) silently injects values the system assumes impossible. Audit by pairing each write-side check with its read-side mirror and treating any unpaired check as a defect.

## CMC-Rd11-1 — Lock-substance preservation contract
- **content:** Every concurrent-writer surface routes through the single helper module `lib/with-lock.cjs`; no ad-hoc inline lock implementations survive. Two mechanisms behind one module, each matched to its surface's write pattern: (a) append-only audit log → kernel O_APPEND via `appendAuditLine(path, line)` (no userspace lock; line cap 4000 bytes under the 4096 PIPE_BUF floor, visible truncation marker on overflow); (b) read-modify-write structured state → `withLock(path, asyncFn)` wx-sentinel-file userspace mutex (exponential backoff 50·2^n ms, 5 attempts, `ELOCKBUSY` error code, stale-lock force-unlink after 60s with stderr warning, release in `finally`). Direct `fs.appendFileSync` against the audit log is "forbidden surface"; consumers cite the helper by path rather than re-implementing.
- **why:** Two surfaces had multiple concurrent writers — an append-only heal audit log (byte interleaving risk) and a work register mutated by three ops (last-write-wins lost-mutation risk). The contract half (this CMC) exists because a correct helper is worthless if the *next* op writes its own inline lock: the preservation rule makes the helper the only legal path.
- **source:** `redesign/cli-spec.v2.md:915-934` (§1.15 complete: anchor at `:921`, mechanisms `:923-925`, consumer wiring + forbidden-surface rule `:927-932`, rationale `:934`); substrate `plugins/essense-flow/lib/with-lock.cjs` (T-951).
- **proposed inline principle:** When a file has multiple concurrent writers, pick one concurrency mechanism per write pattern (kernel-level append atomicity for logs; a userspace mutex for read-modify-write state), put both behind one shared helper module, and make direct writes to those files a banned pattern — the discipline holds only if the helper is the single legal path.
- **keep/fold/drop:** **KEEP.** Generic, public-ready concurrency guidance with concrete mechanism choices and failure-mode rationale. Independent of all four rebuild changes.
